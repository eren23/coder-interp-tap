What does a fine-tune actually learn?

A fine-tune is a routing change, not a concept-addition. The model already had Rust-shaped circuitry. The LoRA promoted it from background to foreground.

Model Qwen2.5-Coder-1.5B Method SAE drift → VPD → integrated gradients Date May 2026

Train a LoRA on a Rust codebase and you expect to find new Rust circuitry inside. We trained that LoRA. We looked. The honest answer is: it didn't add Rust knowledge. Rust was already in the base model. The LoRA changed which of its existing concepts got picked up, how strongly, and on which tokens.

That sounds small. It isn't. The same shift took pub use crate::'s next-token guess from a generic 'error' (3.9%) to the author's own crate name 'config' (26%). The fine-tune rewired prediction without learning anything new — and that gap, between "added knowledge" and "rerouted attention", is what this post is about.

Setup

Base model: Qwen/Qwen2.5-Coder-1.5B, a 28-layer transformer pretrained on a generic code mix. Adapter: rank-32 LoRA, trained for 3 epochs on a personal GitHub corpus — 6,374 files, 17M tokens, 13 Python repos, 3 Rust crates, 3 TypeScript apps, 1 Go service. Training loss 1.05 → 0.62.

We read the resulting model three ways, each more rigorous than the last:

  1. SAE drift. Train a TopK SAE on layer-12 activations and compare which features fire more on BASE versus the LoRA-merged model. Easy, fast, mostly cosmetic.
  2. Variational parameter decomposition. Goodfire's full VPD on the weights — rank-1 decomposition plus a gating network, trained with five losses and a persistent adversarial attack on the masks. Heavy, slow, faithful.
  3. Multi-layer attribution. Eight independent VPDs (layers 0/4/8/12/16/20/24/27), stitched into one system, with integrated gradients on the gates for a specific (prompt, target-token) pair.

The three readings disagree with each other in instructive ways. Each one tells a different story about what the LoRA did, and only the last one survives a serious control.

First read: SAE drift

The cheapest reading. Run a held-out slice of generic code through BASE, then through LoRA-merged, and watch which SAE features change firing rate.

1,848 features had non-trivial activation-rate changes. The 99th percentile of |log-ratio| hit 11.46 — features that fired ~100,000× more or less often after the LoRA merged. We labeled the top movers with DeepSeek-V3:

Out of context, this is the version of the story we almost shipped. Django field detectors. Python package metadata. License boilerplate. Looks like the LoRA bolted a small library of personal coding concepts onto the base model.

But SAE drift measures rate of firing, not causal contribution. If the evaluation slice happens to contain more Rust or more Django than the BASE pretraining distribution, every feature aligned with those will look "newly important" — whether or not the LoRA touched it. We didn't trust the read.

Second read: VPD on the weights

We re-implemented the full Goodfire VPD recipe faithfully. VPD decomposes each targeted weight matrix W into a sum of rank-1 components UVT plus a residual Δ, with a causal-importance network Γ(x) that learns, per token, which components matter. A component is "alive" when its gate fires on a nontrivial fraction of training tokens.

The adversary is the active ingredient

Run VPD without its adversarial inner loop and the gates collapse — every component looks "useful", 99% of gates exceed 0.5. With the persistent-PGD attack on the mask source r, the model is forced to use as few components as possible to defend its own predictions, and the gates sparsify to a few percent. That differential is the entire reason VPD works.

On the Qwen2.5-Coder-1.5B sweep, mean gate magnitudes settle in the 10⁻³ range across all 8 trained layers. That's the regime where "alive" is meaningful.

The gut check that killed the headline

Both writeups would have been clean. "LoRA learned Rust imports." "LoRA learned doc-comment patterns." Reviewers would have nodded.

We ran the control anyway. Take the BASE Coder-1.5B with no LoRA merged, train a fresh VPD on the same corpus, autolabel it with the same prompt. It surfaced the same Rust labels. pub. use. ///. Different component indices in a different model — same labels.

The labels described the corpus, not the LoRA. If your training data is Rust-heavy, the alive components will be Rust-shaped whether or not you fine-tuned.

VPD is a faithful weight decomposition: it finds the directions that matter for reconstructing the model's behavior on the data you train it on. Feed it Rust, it finds Rust-shaped directions. The base model already had them.

The real question is not which components are alive, but how the alive set differs between BASE and LoRA on the same input.

What the LoRA actually did: routing

Holding the corpus fixed, the LoRA-merged model has more alive components, fires them harder, and reroutes attention into them.

None of this is new concepts. It's routing: which directions the model uses, how strongly it gates them, and how much of attention is dedicated to them. The LoRA didn't bolt on Rust knowledge. It promoted Rust knowledge from background to foreground.

Why this matters

If a fine-tune is a routing change rather than a concept addition, two practical consequences follow.

First: the standard story about fine-tunes "teaching" the model is misleading. What you can teach a 1.5B model in a few thousand steps is which of its existing concepts to lean on, in which contexts, with what intensity. New concepts come from pretraining; fine-tunes redistribute attention over them.

Second: routing is observable, individual, and small. Two engineers in the same codebase write code that looks identical at the syntax level but trains different routing deltas. A LoRA on engineer A's PRs against the same base is a candidate signature for how A's code asks the model to think. We are not productizing this here — the model is small, the training is short, the LoRA is on one person. But the methodology composes. A larger model and a longer VPD run on a team's monorepo would give you a per-engineer routing delta you could browse with the same UI as above.

Internal follow-up notes on this direction live in the repo (future-work-notes/developer-fingerprinting.md); they're not public yet because we haven't run the validation experiments.

Honest limitations

  • Scale. Qwen2.5-Coder-1.5B is small. The VPD paper trains for 400,000 steps with a shared transformer Γ over all matrices; we train for 1,500 steps with a per-matrix MLP Γ. Numbers should be read as directional, not absolute.
  • Single individual. The LoRA was trained on one person's 20 repos. No statistical comparison to a population. The fingerprinting claim survives only at case-study level.
  • No causal ablation. The attribution graph is a faithful linearization of the gate-level Jacobian via integrated gradients, but we did not run targeted ablations to confirm that suppressing a highlighted circle actually changes the predicted token. That is the obvious follow-up.
  • The βΔ bug. Early VPD runs had a hardcoded 1e-3 in the main-phase loss that was supposed to be 1e7. We caught it and re-ran; numbers in this post are from the corrected runs.
  • Reconstruction is exact by construction. Final ‖W − ΣUiViT − Δ‖₂ ≈ 0 because Δ absorbs whatever the rank-1 decomposition misses. What's load-bearing is the sparsity of the gates, not the residual.

None of the limitations change the headline. The corpus drives which components look interesting — that's what made us nearly ship a wrong story. The LoRA drives routing — that's the story we ended up shipping.

If you build on top of this, especially in the direction of per-engineer fingerprinting, we'd like to hear about it.
